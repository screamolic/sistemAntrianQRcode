# Phase 2.2: Design and Implement Prisma Schema

**Phase:** 2
**Plan:** 2.2
**Status:** Ready to Execute

---

## Objective

Create Prisma schema for Users, Queues, and QueueEntries with proper relationships.

---

## Tasks

### 1. Define User Model
- [ ] Create `User` model in `schema.prisma`
- [ ] Fields: id, email, name, password, role, createdAt, updatedAt
- [ ] Add unique constraint on email
- [ ] Add indexes for common queries

### 2. Define Queue Model
- [ ] Create `Queue` model
- [ ] Fields: id, name, adminId, isActive, createdAt, updatedAt
- [ ] Relation to User (admin)
- [ ] Unique queue names per admin

### 3. Define QueueEntry Model
- [ ] Create `QueueEntry` model
- [ ] Fields: id, queueId, phoneNumber, name, position, status, notifiedAt, servedAt, createdAt
- [ ] Relation to Queue
- [ ] Enum for EntryStatus (WAITING, SERVED, NO_SHOW)

### 4. Define Enums
- [ ] Create `Role` enum (ADMIN, SUPER_ADMIN)
- [ ] Create `EntryStatus` enum

### 5. Add Indexes
- [ ] Index on User.email
- [ ] Index on Queue.adminId
- [ ] Index on QueueEntry.queueId
- [ ] Index on QueueEntry.phoneNumber
- [ ] Composite index on QueueEntry (queueId, status)

### 6. Run Migration
- [ ] Run `npx prisma migrate dev --name init`
- [ ] Verify migration files created
- [ ] Check database tables created
- [ ] Run `npx prisma generate` to generate client

### 7. Install Prisma Studio (Optional)
- [ ] Run `npx prisma studio` to view data
- [ ] Verify UI opens correctly

---

## Verification Criteria

- [ ] Schema compiles without errors
- [ ] Migration runs successfully
- [ ] Tables created in database
- [ ] Prisma Client generated
- [ ] Relations work correctly
- [ ] Indexes present in database

---

## Notes

- Use `cuid()` for IDs - better than UUID for performance
- Timestamps handled by Prisma with `@default(now())`
- Relations use foreign keys automatically
